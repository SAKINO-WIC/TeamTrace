# TeamTrace 数据库变更规范 v1.0

## 目标

- 确保数据库变更可审计、可回滚、可复现。
- 避免在开发过程中“随手改库”导致前后端与文档失配。

## 1. 变更边界

以下内容视为数据库变更，必须走单独 PR：

- 新增/删除/修改表结构
- 字段类型与约束调整
- 索引、唯一键、外键、检查约束变更
- 与业务语义相关的默认值变更

禁止与前端 UI 改动混在同一个 PR。

## 2. 必须提交的文件

每一次数据库结构变更都必须同时提交：

1. `database/migrations/<yyyymmdd>-<short-desc>.sql`
2. `database/rollback/<yyyymmdd>-<short-desc>-rollback.sql`
3. 变更说明（在 PR 描述中）

## 3. SQL 书写要求

- 脚本可重复执行（尽量使用 `IF EXISTS` / `IF NOT EXISTS`）
- 明确事务边界（适用时）
- 明确影响范围（表、字段、索引）
- 不在迁移脚本中写真实业务数据

## 4. 审核清单

PR 审核时至少检查：

- 是否破坏兼容（旧接口是否还能运行）
- 是否影响已有约束逻辑
- 是否补齐回滚脚本
- 是否同步了 API 文档与系统设计文档

## 5. 执行环境策略

- 开发环境先执行迁移验证
- 预发布环境复验
- 生产执行需提前备份并记录执行人/时间/结果

## 6. Sealos 云数据库特别说明

- 云数据库实例不是版本仓库，SQL 脚本才是版本源。
- 任何在控制台手工执行的结构变更，必须在当日补齐 migration 与 rollback。
