## 一、产品概述

### 1.1 产品名称
- **中文名**：摸鱼终结者  
- **英文名**：TeamTrace  
- **定位**：大学生小组作业贡献管理系统

### 1.2 核心目标
解决大学生小组作业中普遍存在的三大痛点：
- **分工不清**：任务边界模糊，责任不明
- **进度拖沓**：缺乏过程监督，效率低下
- **贡献难量化**：干多干少一个样，公平性缺失

通过**过程留痕、任务拆解、进度可视化、匿名互评、教师评分**等机制，实现“谁干了什么、谁没干什么”全程可追溯、可量化。

### 1.3 适用场景
- 高校课程中的小组作业/项目
- 尤其适用于周期较长（数周至一学期）、需多人协作的任务
- 支持同一教师管理多个班级，同一班级多个任务

## 二、用户角色与权限

| 角色             | 权限说明                                                     |
| ---------------- | ------------------------------------------------------------ |
| **管理员**       | 管理教师/学生账户（增、删、改、查）；生成教师邀请码；查看所有班级数据（只读）；查看操作日志 |
| **教师**         | 创建/管理班级；生成班级邀请码；管理学生分组；发布总任务；开启/关闭互评；评分（个人/小组）；处理申诉；查看报告；可手动重新开启互评 |
| **学生（普通）** | 加入班级；自建/加入小组；认领子任务；提交子任务；参与互评；查看个人/小组进度；发起申诉 |
| **学生（组长）** | 拥有普通学生全部权限；拆解总任务为子任务；指派/取消子任务；审批成员加入（如启用审批模式）；打回子任务；管理小组成员 |

## 三、核心业务流程

### 3.1 注册与登录

| 角色   | 注册方式                       | 说明                                                         |
| ------ | ------------------------------ | ------------------------------------------------------------ |
| 管理员 | 内置固定账号                   | 不可通过注册获得                                             |
| 教师   | 手机号 + 密码 + **教师邀请码** | 邀请码由管理员生成，有效期30天，使用后失效；管理员可撤销未使用的邀请码 |
| 学生   | 手机号 + 密码                  | 不填写邀请码，默认为学生角色                                 |

### 3.2 班级管理（教师）

**创建班级**
- 必填：班级名称、学期
- 小组人数不固定：至少 1 人即可成组，一人成组时互评不适用
- 系统自动生成班级编号：`年份 + 4位自增序号`（如20250001），全局唯一

**班级邀请码**
- 教师手动生成，一次性使用，有效期24小时
- 可随时刷新：旧邀请码立即失效，生成新码
- 每个班级独立邀请码

**解散班级**
- 教师主动解散 → 数据软删除（`is_deleted=1`）
- 30天内可恢复（恢复后需重新生成邀请码）
- 30天后物理删除

**管理学生**
- 查看学生列表
- 手动移除学生（移除后该学生数据保留至学期结束）
- 锁定分组（锁定后学生不可自建分组）

### 3.3 加入班级（学生）
- 学生通过班级邀请码加入班级
- 一个学生可加入多个班级

### 3.4 分组管理

#### 两种分组方式

| 方式             | 操作者       | 说明                                                         |
| ---------------- | ------------ | ------------------------------------------------------------ |
| **教师手动分组** | 教师         | 教师直接指定小组成员及组长。分组后可锁定，学生不能再自建     |
| **学生自建分组** | 学生（组长） | 学生创建小组 → 自动成为组长 → 生成小组邀请码。组长可选加入模式：<br>• 直通模式：输入邀请码直接加入<br>• 审批模式：需组长审批通过 |

**小组邀请码**

**小组邀请码**

- 学生自建分组时自动生成。
- 教师手动创建的分组 invite_code 可为空（NULL），由后端决定是否生成。
- invite_code 字段允许 NULL，NULL 值可重复，只有非 NULL 值强制唯一。
- 有效期由组长或系统自定义，默认 24 小时。
- 组长可刷新邀请码（旧码立即失效，新码生成）。

**一人成组**
- 允许。一人成组时，互评环节不开启。

**小组管理（组长权限）**
- 移除成员（无需填写原因，无申诉机制）
- 学生不可主动退出小组，只能由教师、管理员或组长操作移除
- 被移除成员数据保留至学期结束

**教师移动成员**
- 教师可将学生从一个小组直接移动到另一个小组
- 系统自动完成：从原小组移出 + 加入新小组

### 3.5 任务管理

#### 3.5.1 教师发布总任务

**必填字段**
- 任务名称
- 描述
- 总体截止时间

**互评设置**
- 是否开启互评：是/否
- 若开启：
  - 互评时长（单位：小时，默认1小时）
  - 互评截止时间 = 任务截止时间 + 互评时长
  - 教师可手动修改互评截止时间（**只能延后**）
  - 互评截止后自动关闭互评入口，但教师可**手动重新开启互评**（供忘记互评的学生补评）
- 若不开启：无互评环节

#### 3.5.2 组长拆解子任务

**操作权限**：仅组长

**子任务字段**
- 名称
- 描述
- 质量要求（可选）
- 截止时间
- 负责人（可指派给具体组员，或留空待认领）

**后续操作**
- 组长可修改/删除已发布的子任务
- 组长可重新指派已被认领的子任务（原认领人状态变更为“已取消认领”，并发送通知）
- 组长可取消组员的认领（发送通知）

**进度计算规则**
- 采用平均权重：`进度 = 已完成子任务数 / 总子任务数`
- 个人进度：仅计算本人认领的子任务（`本人完成数 / 本人认领总数`）
- 小组进度：计算小组所有子任务（`小组完成数 / 小组总子任务数`）
- 组长本人也拥有个人进度（组长自己承担的子任务）

#### 3.5.3 子任务执行与审批

**认领**
- 组员可自行认领未分配的子任务
- 组长也可直接指派

**提交**
- 组员完成后提交，可上传附件（图片、文档、链接）

**审批**
- 组长可查看提交内容，选择“通过”或“打回修改”
  - **通过**：子任务状态变为“已完成”，计入进度
  - **打回**：子任务状态变为“进行中”，不计入进度，组员需修改后重新提交
- **组长审批不是必经环节**：组员提交后进度即已体现；组长仅在发现问题时打回。因此**无超时自动通过机制**。

**逾期处理**

- 子任务超过截止时间未完成 → 自动标记 is_overdue = 1。
- 逾期标记与业务状态（status）完全解耦，逾期后仍可提交和审批。
- 子任务状态定义为：1 待认领、2 进行中、3 待审批、4 已完成。

### 3.6 进度可视化

| 视图         | 计算范围                            | 更新方式              |
| ------------ | ----------------------------------- | --------------------- |
| **个人进度** | 本人认领的子任务（完成数/认领总数） | 实时（WebSocket推送） |
| **小组进度** | 小组所有子任务（完成数/总子任务数） | 实时                  |
| **教师看板** | 班级内所有小组/成员的进度           | 实时                  |

**触发进度更新的操作**
- 组员提交子任务
- 组长打回子任务
- 组长通过子任务

### 3.7 互评机制

**开启条件**
- 教师发布总任务时选择“开启互评”，并设置互评时长（小时）
- 互评在任务截止后**自动开启**
- 到达互评截止时间后**自动关闭互评入口**，但教师可**手动重新开启**

**匿名性**
- 学生端：完全匿名（看不到评价人身份）
- 教师端：可见评价人（用于处理申诉）

**评分方式**
- **教师自定义评分范围**，默认为 **0-100分**（整数或一位小数）
- 可填写评语（文本，可选）

**未提交处理**
- 截止后未提交互评的成员 → 互评分数记为 **NULL**
- 报告中显示“未参与互评”
- 教师不可手动补充评价，但可以**重新开启互评**（延长互评时间）让忘记的人补评

### 3.8 评分体系

#### 3.8.1 每次任务评分

| 评分类型 | 评分人 | 存储位置                                 | 说明                 |
| -------- | ------ | ---------------------------------------- | -------------------- |
| 个人评分 | 教师   | `teacher_scores` (target_type='student') | 教师对每个学生的评分 |
| 小组评分 | 教师   | `teacher_scores` (target_type='group')   | 教师对每个小组的评分 |

**权重设置**

- 教师可在任务中设置互评分数占比与教师评分占比。
- 系统通过数据库 CHECK 约束确保两者之和在合理范围内（0.999 ~ 1.001）。

#### 3.8.2 课程总评分

- **自动计算**：教师选择各任务成绩按权重加权汇总
- **手动打分**：教师直接为每个学生/小组打分
- 存储于 `course_scores` 表

**修改重算**
- 教师修改任何评分（任务评分、互评、权重）后，系统自动重新计算相关报告和课程总评分

### 3.9 申诉流程

**可申诉事项**
- 子任务被误判（如打回不合理）
- 互评分数异常（明显不公）
- 教师评分不公

**申诉材料**
- 学生填写理由，可上传附件（图片、文档、链接）

**处理流程**
1. 学生提交申诉
2. 教师查看申诉内容
3. 教师给出处理结果（同意/驳回）
4. 系统发送站内通知学生结果

**限制**
- 每个学生对同一任务最多申诉 **3次**

**申诉后调整**
- 教师可根据申诉结果修改评分或子任务状态

### 3.10 站内通知

**触发点**
| 事件                      | 说明                     |
| ------------------------- | ------------------------ |
| 子任务即将截止            | 剩余时间 ≤ 任务时长的10% |
| 组长打回子任务            | 通知对应组员             |
| 互评开启                  | 通知小组所有成员         |
| 互评即将截止              | 剩余时间 ≤ 互评时长的10% |
| 申诉结果通知              | 教师处理后               |
| 新成员加入小组            | 通知组长和所有成员       |
| 子任务被重新指派/取消认领 | 通知原负责人             |

**用户操作**
- 标记已读
- 批量删除

### 3.11 数据保留与清理

**软删除**
- 核心业务表（users、classes、class_students、tasks、groups、subtasks、peer_reviews、teacher_scores、course_scores、reports、appeals 等）均包含 is_deleted 和 deleted_at。
- 日志类表（notifications、operation_logs）及邀请码表（teacher_invite_codes、class_invite_codes）不使用软删除，采用定期清理策略。

**物理删除**
- 定时任务（每天执行）：删除 `deleted_at` 超过 **30天** 的记录

**班级解散**
- 教师主动解散 → 立即软删除
- 30天内可恢复（恢复后需重新生成邀请码）
- 30天后物理删除

**学生账号生命周期**
- 学生可主动注销
- 管理员可随时清除
- 账号最多保留 **4年**（从注册日起算）

### 3.12 操作日志

**记录范围**
- 创建/修改/删除班级
- 创建/修改/删除任务
- 创建/修改/删除小组
- 创建/修改/删除子任务
- 成员增删（加入/移除小组）
- 评分修改（教师评分、互评）
- 申诉处理

**日志字段**
- 操作人ID
- 操作类型（CREATE/UPDATE/DELETE）
- 目标类型（class/task/group/subtask/...）
- 目标ID
- 变更前后值（JSON格式）
- IP地址
- UserAgent
- 操作时间

### 3.13 并发控制

- 在可能并发冲突的表（`subtasks`, `group_members` 等）中增加 `version` 字段
- 使用**乐观锁**：更新时检查版本号，不一致则重试

### 3.14 定时任务清单

| 任务名称           | 执行频率 | 说明                                                         |
| ------------------ | -------- | ------------------------------------------------------------ |
| 子任务提醒         | 每小时   | 检查即将到期的子任务（剩余时间 ≤ 总时长的10%），发送站内通知 |
| 互评开启提醒       | 实时触发 | 互评开启时立即发送通知                                       |
| 互评即将截止提醒   | 每小时   | 互评剩余时间 ≤ 总时长的10%时发送提醒                         |
| 互评自动关闭       | 每小时   | 检查已过互评截止时间的任务，关闭互评入口                     |
| 软删除数据物理删除 | 每天     | 删除已软删除超过30天的记录                                   |
| 邀请码清理         | 每天     | 删除已过期且未使用的邀请码（教师邀请码、班级邀请码、小组邀请码） |

## 四、非功能需求

### 4.1 实时性
- 进度看板使用 **WebSocket** 实现实时更新（延迟 ≤ 2秒）

### 4.2 安全性
- 认证：JWT（有效期7天，可刷新）
- 权限中间件：
  - 学生只能操作自己所在班级/小组的数据
  - 教师只能操作自己创建的班级
  - 组长只能操作自己小组的数据
- 密码加密：bcrypt（强度10）
- 防SQL注入：参数化查询

### 4.3 可扩展性
- 预留“分工模板库”功能（`task_templates` 表），后期教师可基于模板快速创建任务

### 4.4 可维护性
- 操作日志完整，便于问题追溯
- 所有表包含 `created_at` / `updated_at` 时间戳
