# TeamTrace 屏幕级交互设计 v1

> **本文档回答的问题是：用户打开系统后，每一秒看到什么、能点什么、点了之后发生什么。**
> 
> 设计原则：**用户永远不需要思考"下一步该干什么"——系统必须主动告诉他。**

---

**每个可点击元素的行为**：

| 用户点击 | 发生什么 |
|---------|---------|
| 🔴 "待处理申诉 3件" 卡片 | → Screen 2.1（申诉中心，自动筛选"待处理"） |
| ⚠️ "即将截止 2个" 卡片 | → 跳转到那个快截止任务的进度页 |
| `[处理申诉 →]` | → Screen 2.1 |
| `[查看进度 →]` | → 跳转到数据结构B班 → 作业2 → 进度 Tab |
| 班级卡片"软件工程A" | → Screen 2.2（班级详情） |
| 侧边栏"班级列表" | → 班级列表页 |
| 侧边栏"评分中心" | → 评分中心 |
| 侧边栏"申诉中心" | → 申诉中心 |

**为什么这样设计**：
- **待处理申诉是最高优先级**：红色 + 最上面 + 展开显示前三条
- **即将截止次之**：黄色 + 显示剩余天数
- **班级列表最下面**：因为教师日常最频繁的操作是"进入某个班级"，不是"看班级列表"
- 每个卡片都是可点击的**快捷入口**，不需要教师记住菜单在哪

**Tab 切换逻辑**：

| Tab | 显示内容 | 数据来源 |
|-----|---------|---------|
| 📋 概览 | 任务描述 + 各组进度卡片 + 快捷操作 | `GET /teacher/classes/:cid/tasks/:tid` |
| 📊 进度 | 组选择器 + 该组成员子任务进度表 | `GET .../groups/:gid/subtasks/progress` |
| ⭐ 互评 | 组选择器 + 该组互评记录表 | `GET .../groups/:gid/peer-reviews` |
| 📈 成绩 | 组选择器 + 该组成绩汇总 + 教师打分入口 | `GET .../groups/:gid/score-summaries` |
| 📢 申诉(3) | 该任务的申诉列表 + 处理弹窗 | `GET .../appeals` |

**关键设计：Tab 标签上的数字** — `申诉(3)` 直接告诉教师"这里有 3 条申诉等着你"，不需要点进去才知道。

**组选择器的行为**：
- 所有 Tab 共用同一个组选择器（在 Tab 栏下方）
- 切换组时，当前 Tab 内容自动刷新
- 切换到另一个 Tab 时，组选择器保持在之前选的组

# 第四部分：全局交互规则

### 规则1：每页必须有"主行动按钮"

| 页面状态 | 主行动 |
|---------|--------|
| 教师工作台（空） | [创建第一个班级] |
| 教师工作台（有申诉） | [处理申诉 →] |
| 学生首页（空） | 邀请码输入框 + [加入班级] |
| 学生首页（有待办） | [去提交 →] / [去互评 →] |
| 班级详情（无任务） | [发布第一个任务] |
| 任务详情（未打分） | 打分入口高亮 |

### 规则2：操作后必须有"下一步引导"

每个操作成功后，不直接跳走——而是：
1. 页面内反馈（成功提示 + 数据刷新）
2. 显示"下一步建议"（如发布任务后 → "你是否要管理分组？"）

### 规则3：错误状态必须有"怎么解决"

不是只显示"操作失败"，而是：
- "邀请码已过期" → "请联系老师重新生成邀请码"
- "密码格式不正确" → "密码需8-20位，包含字母和数字"
- "网络错误" → "请检查网络连接后重试 [重试]"

### 规则4：Tab 切换保留上下文

- 组选择器的选择状态在所有 Tab 间共享
- 筛选条件在切换 Tab 后保留
- 返回上一页时，恢复之前的滚动位置和 Tab 选择
